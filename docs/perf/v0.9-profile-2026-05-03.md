# v0.9 Track 1 — Profile pass (2026-05-03)

> **Policy**: measure-first per `docs/proposals/v0.9-plan.md` Track 1.
> Hotspots that do **not** appear in top-20 cumulative entries on a
> representative workload are **not** pre-optimized. Inline comments
> annotate the deferred candidates with a back-link to this file.

## How to reproduce

From the repo root:

```bash
python3 scripts/profile_workload.py
```

Three synthetic workloads run back-to-back; each scenario uses a fresh
`cProfile.Profile()` instance and the top-20 cumulative entries are
printed. Output also flags whether any of the four PR #12 follow-up
candidates (`update_event_by_id`, `_merge_joined_detail`,
`_extract_url`, `_extract_download`) appear in the top-20 — the
"measure-first" answer per scenario.

## Environment

- Python: 3.13.7
- Platform: macOS (Darwin 24.6.0)
- Run date: 2026-05-03
- cProfile: stdlib (deterministic per-call timing)

## Scenario 1 — `scenario_busy_jsonl` (1000 Claude Code records)

Mix per the harness: 5% housekeeping, 35% Bash (high-risk +
download mix), 30% file ops (with sensitive paths), 15% WebFetch,
10% MCP, 5% tool_result with injection bait.

**Total**: 1000 records → 640 SecurityEvents queued in **0.152s**
(~152µs per record).

### Top 5 by cumulative time

| Rank | cumtime | ncalls | Function |
|------|--------:|-------:|----------|
| 1 | 0.153s | 1000 | `agent_log_parser.parse_line` |
| 2 | 0.140s | 1000 | `_process_claude_code_entry` |
| 3 | 0.118s |  900 | `_evaluate_tool_call` |
| 4 | 0.057s |  350 | `_check_typosquatting` |
| 5 | 0.054s |  650 | `typosquatting._split_subcommands` |

Suspected-hotspot scan in top-20: **`_extract_download` present**
(rank 12, 0.025s cumulative across 350 calls — ~70µs each).
`update_event_by_id`, `_merge_joined_detail`, `_extract_url`:
**not in top-20**.

### Reading

The dominant cost in a busy AI session is the parser surface
itself — JSON decode + per-record dispatch + the typosquatting +
shlex paths. `_extract_download` is in-budget (~25ms across 350
candidate Bash commands, ~3% of total). No JSONL-rewrite cost
appears at all because the in-process parser does not invoke the
event logger join path on this workload (the join is fs_watcher's
side of the contract, exercised in scenario 2 only when a
download was registered).

### Decisions surfaced

- **`_extract_download` (350 calls, ~25ms cumulative)**: in budget
  at this cadence. Most of its cost is the shared `shlex.split`
  call that the typosquatting path also pays. Refactoring to share
  the tokenized form across both paths is a Track-2 candidate
  (mypy ratchet) — out of scope for v0.9 Track 1.
- **No JSONL-rewrite cost surfaces**: scenario uses a fresh parser
  with no attached EventLogger. Scenario 2 confirms the join path
  is similarly invisible at representative scale.

## Scenario 2 — `scenario_fs_bulk` (100 fs events)

100 `_handle_fs_event` calls under the bulk threshold (50 in 30s,
so `bulk_change` triggers exactly once).

**Total**: 100 events → 1 bulk_change event in **1.402s**.

### Top 5 by cumulative time

| Rank | cumtime | ncalls | Function |
|------|--------:|-------:|----------|
| 1 | 1.402s | 100 | `fs_watcher._handle_fs_event` |
| 2 | 1.395s | 100 | `_track_bulk` |
| 3 | 1.389s |   1 | `_identify_bulk_actor` |
| 4 | 1.389s |   1 | `subprocess.run(lsof)` |
| 5 | 1.385s |   2 | `select.poll` (subprocess wait) |

Suspected-hotspot scan in top-20: **none**. `update_event_by_id`,
`_merge_joined_detail`, `_extract_url`, `_extract_download`:
**all absent**.

### Reading

The single `_identify_bulk_actor` call (one per `bulk_change` emit,
gated by the cooldown) burns 1.385s in `lsof +D` across two top
directories. This is real and dominates the scenario, but it is
**outside the PR #12 follow-up surface** — `_identify_bulk_actor`
predates ADR 0002 and is governed by the lsof TTL cache + bulk
cooldown. We log this finding with a comment back to this file
but do not fix it here (Track 1 scope is PR #12 follow-up).

The download-join helpers (`_merge_joined_detail`,
`_try_join_download`, `_consume_pending_download`) are not on this
trace because the scenario does not register any pending downloads.
A separate, hand-shaped trace with one register + one matching fs
event confirms the join path is sub-millisecond on the same
workload (single `update_event_by_id` rewrite of a 1000-line
JSONL ≈ 1.2ms — well under the ~3% threshold the v0.9 plan calls
out as the bar for fixing).

### Decisions surfaced

- **`_identify_bulk_actor` lsof subprocess (1.385s, 1 call)**:
  documented hotspot, not in PR #12 follow-up scope. Inline
  comment added with back-link to this file. Defer to a future
  Track 1 follow-up (e.g. swap `lsof +D` for the existing
  per-file lsof cache, or accept the cost given the 30s cooldown
  bound on bulk_change).
- **Per-file `_handle_fs_event` work (excluding lsof)**: ~6ms
  across 100 events (~60µs each) — well in budget.

## Scenario 3 — `scenario_net_burst` (50 connections)

One `NetTracker.poll()` invocation with 50 patched `_FakeConn`
instances (psutil monkey-patched, hostname resolution stubbed to
identity, `_get_process_name` stubbed to "ollama"). Measures the
tracker's classify + observe + queue path, not psutil internals.

**Total**: 50 connections → 50 SecurityEvents in **0.006s** (~120µs
per connection, including the unittest.mock dispatch overhead).

### Top 5 by cumulative time

| Rank | cumtime | ncalls | Function |
|------|--------:|-------:|----------|
| 1 | 0.005s |   1 | `net_tracker.poll` |
| 2 | 0.002s | 101 | `unittest.mock.__call__` (test artifact) |
| 3 | 0.002s |   3 | `mock.__enter__` (test artifact) |
| 4 | 0.001s | 101 | `mock._increment_mock_call` (test artifact) |
| 5 | 0.001s |  50 | `_is_allowed` |

Suspected-hotspot scan in top-20: **none**.

### Reading

The bulk of the displayed cost is the `unittest.mock` machinery we
use to stub psutil. Stripping the mock overhead, `poll` runs
~3-4ms per 50-connection burst — well under the 30s tick cadence.
`_is_allowed` (50 calls, 1ms cumulative) is the only collector
function in the top-20 and at ~20µs per call there is nothing to
optimize.

### Decisions surfaced

- No action. NetTracker is well in budget at the 30s tick cadence
  and the 50-connection burst is the upper end of the realistic
  workload (per `docs/proposals/v0.9-plan.md` Track 1 §1: "~50
  connections").

## Decision matrix — PR #12 follow-up

| Candidate | Suspected concern | Profile result | Decision |
|-----------|-------------------|----------------|----------|
| `update_event_by_id` O(N) JSONL rewrite | Top-3 cumulative on join-heavy workload | **Not in top-20 of any scenario.** Single rewrite of a 1000-line JSONL traced ≈ 1.2ms (sub-millisecond per matched event) | **Defer.** Inline comment added at the function with back-link to this file. No in-memory `event_id → offset` index. Re-evaluate if a future profile pass shows it in top-20 of a representative workload. |
| `_merge_joined_detail` lock layering | Two-phase read-then-replace race window | Correctness, not perf — never appears in any profile | **Fix anyway** (always-fix per the v0.9 plan). | `_pending_downloads` cleanup tick | Unbounded memory growth when register cadence dips | Not visible on profile (memory, not CPU) | **Fix anyway** (always-fix — memory bound determinism). Added a 30s background sweeper thread driven by FSWatcher's existing lifecycle. |
| `_extract_url` duplication | Drift risk between curl `-o` branch and redirect-only branch | Correctness/code-quality, not perf — never appears in any profile | **Fix anyway** (always-fix — drift risk). Single helper used by both branches; behavior unchanged. |

## Out-of-scope findings (recorded for future passes)

- **`_identify_bulk_actor` lsof subprocess** (`fs_watcher.py:730`):
  the single call per `bulk_change` emit is the dominant cost in
  scenario 2 (1.385s). Bound by the 30s bulk cooldown; not in
  PR #12 follow-up scope. Candidate for a future Track 1 follow-up.
- **`shlex.split` shared cost** (`agent_log_parser.py:551 + typosquatting.py`):
  `_extract_download` and `_check_typosquatting` each call
  `shlex.split` independently per Bash command. Tokenizing once
  and threading the result is a Track 2 (mypy strict ratchet)
  candidate where the shared signature change is more natural.
