# v0.11 — `_identify_actor` lsof cache fix

**Status**: Implemented (PR coming).
**Date**: 2026-05-05.
**Scope**: `sentinel_mac/collectors/fs_watcher.py::FSWatcher._identify_actor`.
**Reproducer**: `python3 scripts/profile_lsof.py`.

## Background

The v0.9 Track 1 profile pass deliberately patched `_identify_actor`
out (see `docs/perf/v0.9-profile-2026-05-03.md`) to measure the
watcher logic in isolation, recording lsof as a separate concern with
an estimated cost of "1.4s/100 events" for a future investigation.
This document is that investigation.

## Method

`scripts/profile_lsof.py` exercises three burst patterns against the
real macOS `lsof` subprocess. All paths are inside a fresh `tmp` and
marked sensitive so `_identify_actor` is actually invoked.

| Scenario | Path mix | Why this pattern matters |
|---|---|---|
| `unique` | 100 different files | Worst case: build / install creates many fresh files |
| `repeat` | one file × 100 calls | Cache must absorb fully (best case) |
| `mixed`  | 10 files × 10 calls each | Typical IDE edit cycle |

Test machine: macOS, Python 3.13.7, daemon idle.

## Findings

The pre-fix cache had two bugs that made it functionally a no-op for
every realistic workload:

1. **Negative results were not cached.** The cache only stored entries
   where lsof returned a holder (`if result.stdout.strip():`). Most
   fs events land on files with no holder (write-then-close patterns),
   so the cache was empty after every call — `cache_size=0` after a
   100-call burst.
2. **Single global TTL timestamp.** `_lsof_cache_time` was bumped on
   every successful lsof invocation, and the TTL window was checked
   against that single timestamp for all paths. This entangled cache
   freshness across unrelated paths.

Per-call cost (macOS `lsof -F pcn -- <path>`): ~290–340ms. The
2-second TTL was set assuming Linux-grade speed; it is too tight for
even a 10-path rotation to complete a round before the first entry
expires.

### Baseline (pre-fix)

```
unique  : 100 calls, 100 unique paths, elapsed=33556.9ms, per-call=335.57ms, cache_size=0
repeat  : 100 calls, 1 unique paths,   elapsed=33867.6ms, per-call=338.68ms, cache_size=0
mixed   : 100 calls, 10 unique paths,  elapsed=35998.7ms, per-call=359.99ms, cache_size=0
```

The daemon paid ~33s of stalled main-thread time per 100-event burst
on sensitive files. (For a real-world AI-coder install firing dozens
of `pip install` writes, this maps to a multi-second UI hitch.)

### After fix

```
unique  : 100 calls, 100 unique paths, elapsed=29597.7ms, per-call=295.98ms, cache_size=100
repeat  : 100 calls, 1 unique paths,   elapsed=  278.9ms, per-call=  2.79ms, cache_size=1
mixed   : 100 calls, 10 unique paths,  elapsed= 2877.7ms, per-call= 28.78ms, cache_size=10
```

| Scenario | Before | After | Speedup | Reduction |
|---|---:|---:|---:|---:|
| `unique` | 33,557 ms | 29,598 ms |  1.13× | 12 % |
| `repeat` | 33,868 ms |    279 ms | **121×** | **99 %** |
| `mixed`  | 35,999 ms |  2,878 ms |  **12.5×** | **92 %** |

`unique` is bound by the fundamental `lsof` per-file cost — caching
cannot help when no path repeats. The remaining ~30s requires Phase 2
(directory-batch lsof; see follow-up) to address.

## Implementation

The fix replaces the single-timestamp dict with `OrderedDict[str,
tuple[int, str, float]]`:

- Per-entry timestamp inside the value tuple.
- LRU eviction at `_lsof_cache_max = 512` (well above any realistic
  burst).
- Both positive and negative results stored under the same TTL, so
  the common "no holder" case actually benefits from caching.
- TTL bumped from **2s to 30s**. lsof on macOS is slow enough that
  any tighter TTL leaves the cache useless. 30s is well within the
  "actor attribution can be mildly stale" tolerance for fs_watcher,
  which is best-effort by design.

Lines changed: ~25 in `fs_watcher.py`. Public API unchanged.

## Known follow-ups (deferred)

- **Phase 2 — directory-batch lsof.** When a bulk burst is detected
  on the same parent directory, a single `lsof +D <dir> -F pcn` call
  covers every file. Drops `unique` from N×290ms to ~290ms total.
  Requires deferring per-file actor identification, which changes
  event-shape contracts; needs an ADR.
- **psutil alternative.** `psutil.process_iter(['open_files'])` could
  remove the subprocess overhead but costs a per-process syscall on
  macOS and may not be faster in aggregate. Worth measuring if the
  Phase 2 batch approach also turns out to be slow.

## Reproduction

```
python3 scripts/profile_lsof.py
```

Numbers vary with system load — for clean comparisons, run with no
other heavy processes and quit the menu bar app to free its file
handles.
