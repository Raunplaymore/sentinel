## 권장 모양

**테마**: "Polish + Performance" — v0.8에서 명시적으로 미룬 long-tail
TODO 청산 + 사용자 입장에서 "세심한 마무리"로 읽히는 작은 UX wins
묶음. 새 큰 기능 없음. 다음 큰 결정(아래 Tier 4)은 별 대화.

**이 모양을 권장하는 이유**:
- v0.8이 17 PR로 사용자 가시 기능 7개를 쌓았다. 많다. 바로 다음
  큰 테마(dashboard, cross-platform, Pro)로 넘어가면 결함이
  발견되고 코드 품질 부채를 갚는 자연스러운 cooldown 사이클을
  건너뛰게 됨.
- v0.8의 여러 ADR이 명시적으로 v0.9 follow-up을 명시했다 (ADR
  0005 §D5 read-side lock — 이미 PR #22에서 처리됨, 그러나
  "deferred → 다음 minor" 패턴은 ADR 0006 §D5, ADR 0007 §D7
  등에 계속 살아있음).
- v0.8 마무리 시점 사용자 시그널: "팀단위 유료플랜은 뒤로 하고
  서비스 성능 자체에 집중. UX 제공을 고민" — Tier 1 + Tier 2가
  이걸 직접 답한다. Tier 4 (Pro)는 명시적으로 미룸.

**전체 추정**: ~3주 (단일 작업자 기준), ~5 PRs.

## 트랙

### Track 1 — Performance + measure-first (Tier 1A)

ADR 불필요. 측정 먼저, 그 다음 진짜 핫스팟만 fix.

1. **프로파일 패스**: 대표 워크로드를 instrument 또는 `cProfile`로
   돌림 (busy AI session: 분당 ~100 events JSONL, fs_watcher
   bulk_change spike, NetTracker 30s tick에 ~50 connections).
   상위 3개 hot path 식별.
2. 이제 ~6개월 된 **PR #12 follow-up** fix:
   - `_merge_joined_detail` lock 계층 정리 (현재 2단계
     read-then-replace; single-daemon 가정상 race-tight지만
     아키텍처상 noisy).
   - `update_event_by_id` O(N) JSONL rewrite — 프로파일 결과
     상위 3에 들어가면 in-memory event_id → line offset 인덱스
     추가. 그렇지 않으면 "monitor" 주석만 남기고 그대로 둠.
   - `_pending_downloads` cleanup tick — 현재 register 시점에
     inline GC; 30s background sweep으로 전환해서 register 빈도
     무관하게 메모리 bound가 결정적이 되게.
   - curl 분기 사이의 `_extract_url` 중복 제거.

**LOC**: ~400, 대부분 테스트.
**리스크**: 중간 — event_logger lock surface 만짐; Track 1c
read-side lock 테스트 회귀 조심.

### Track 2 — mypy strict ratchet + ruff B/UP (Tier 1B, Q6)

합의된 v0.8 → v0.9 ratchet (Q6 = a). 리뷰 surface 관리 위해
서브 PR 3개로 분할:

- **Track 2a — ruff B (bugbear) + UP (pyupgrade)**: 룰셋 활성화,
  모듈별로 violation fix. v0.6 Track C에서 임시로 깐 per-file
  ignore도 가능한 곳은 정리. ~150 LOC, 대부분 기계적.
- **Track 2b — mypy `check_untyped_defs = true`**: 미타입 함수
  body가 surface됨. 점진적으로 type hint 추가; 현재
  `[[tool.mypy.overrides]] ignore_errors = true` 뒤에 숨어있는
  4 모듈(`core`, `menubar_app`, `agent_log_parser`, `fs_watcher`)이
  서브 PR 하나당 하나씩 override를 잃기 시작.
- **Track 2c — mypy `disallow_untyped_defs = true`**: 최종 ratchet.
  이 시점이면 override가 다 비어있어야.

**LOC**: 서브 PR 셋 합쳐 ~600.
**리스크**: 중간 — type 에러가 실 버그를 surface시킴; follow-up
패치 1-2개 budget.

## v0.9 freeze 시점에 작성할 ADR

새 ADR 2개 + mini-amendment 1개 예상.

- **ADR 0008 — Notification Context Level** (Track 3,
  `notifications.context_level` 토글). 3 값, 디폴트(=`standard`),
  채널별 렌더링 룰, ADR 0007 §D7의 "git.remote는 audit-log-only
  forever" 디폴트 수정 명시.
- **ADR 0009 — Backup Retention Policy** (Track 3,
  `--cleanup-backups --keep N`). cleanup contract, 디폴트 `--keep`
  값, idempotency, dry-run 플래그.
- **ADR 0001 (HostContext) mini-amendment**: ProjectContext의
  mtime invalidation 동작에 대한 D2 reload-table 행. supersede
  아님 — additive.

Track 1 (perf)과 Track 2 (mypy ratchet)는 새 ADR 불필요 — 기존
contract 위 순수 코드/도구 작업.

## v0.9 시작 대화 시 결정할 항목

§B에서 정확히 **하나** 골라서 Track 3 시작 전에 결정.

- **B1**: Track 3에 **Cursor / VS Code Continue 세션 메타데이터**
  (Tier 3 §I)를 포함할까? ~200 LOC + 파서 모듈 추가; 그 시점에
  Cursor 사용자가 요청한 적 있는지 여부에 따라. 디폴트: 요청 없으면
  v0.9에선 skip.
- **B2**: Track 3에 **net_tracker 세션 attribution via cross-collector
  correlation** (ADR 0007 §D5 follow-up)을 포함할까? ~250 LOC + 작은
  ADR. 네트워크 forensics win이지만 Tier 4 dashboard 작업이 audit log를
  분석적으로 소비하기 시작해야 의미가 큼. 디폴트: v0.9에선 skip,
  Tier 4 시작 시 재검토.
- **B3**: v0.9에 **cross-event correlation** primitive (Tier 3 §H)를
  추가할까? "이 typosquat 명령이 30초 전에 unknown-host network
  연결 직후에 일어났다" — 보안 가치 큼, ~500 LOC, in-memory window
  vs SQLite 질문에 새 ADR 필요. 디폴트: v0.9에선 skip, Tier 4
  dashboard 결정과 함께 재검토.

위 디폴트들은 v0.9를 가볍게 유지: Tracks 1 + 2 + 3만. 3주의 집중
작업으로 끝나고 사용자가 v0.10에서 어느 Tier 4 방향 가든 깨끗한
인계.
