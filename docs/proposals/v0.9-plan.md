> **Status**: draft. Not yet committed to. Confirm scope before

> **Author note**: written at the end of the v0.8.0 release work.
> v0.8.0 just shipped (PR #23, tag `v0.8.0`); this document captures

## Recommended shape

**Theme**: "Polish + Performance" — finish the long-tail of TODOs
that v0.8 explicitly deferred, plus a bundled set of small UX wins
the user reads as "thoughtful housekeeping". No new big-bet
features. The next big-bet decision (Tier 4 below) is a separate
conversation.

**Why this shape**:
- v0.8 shipped 7 user-visible features in 17 PRs. That's a lot.
  Going straight into another large-scope theme (dashboard,
  cross-platform, Pro) skips the natural cooldown where defects
  get found and code-quality debt gets paid.
- Several v0.8 ADRs explicitly named v0.9 follow-ups
  (ADR 0005 §D5 read-side lock — already done in PR #22, but the
  pattern of "deferred → next minor" continues for ADR 0006 §D5,
  ADR 0007 §D7, etc.).
- The user signal at v0.8 close was "팀단위 유료플랜은 뒤로 하고
  서비스 성능 자체에 집중. UX 제공을 고민" — Tier 1 + Tier 2
  here directly answers that. Tier 4 (Pro) is explicitly deferred.

**Estimated total**: ~3 weeks (single contributor), ~5 PRs.

## Tracks

### Track 1 — Performance + measure-first (Tier 1A)

ADR-free. Measure first, then fix the actual hotspots.

1. **Profile pass**: instrument or `cProfile` a representative
   workload (busy AI session: ~100 events/min in JSONL, fs_watcher
   bulk_change spike, NetTracker 30s tick with ~50 connections).
   Identify top-3 hot paths.
2. Fix the **PR #12 follow-ups** that are now ~6 months old:
   - `_merge_joined_detail` lock layering (currently 2-step
     read-then-replace; race-tight under single-daemon assumption
     but architecturally noisy).
   - `update_event_by_id` O(N) JSONL rewrite — if profile shows
     this in the top-3, add an in-memory event_id → line offset
     index. Otherwise leave alone with a "monitor" comment.
   - `_pending_downloads` cleanup tick — currently inline GC on
     register; move to a 30s background sweep so memory bound is
     deterministic regardless of register frequency.
   - `_extract_url` duplication between curl branches.

**LOC**: ~400, mostly tests.
**Risk**: medium — touching event_logger lock surface; needs
careful regression on Track 1c read-side lock test.

### Track 2 — mypy strict ratchet + ruff B/UP (Tier 1B, Q6)

The agreed v0.8 → v0.9 ratchet (Q6 = a). Three sub-PRs to keep
review surface manageable:

- **Track 2a — ruff B (bugbear) + UP (pyupgrade)**: enable rule
  sets, fix violations module by module. Per-file ignores from
  v0.6 Track C cleaned up where possible. ~150 LOC, mostly
  mechanical.
- **Track 2b — mypy `check_untyped_defs = true`**: surface untyped
  function bodies. Add type hints incrementally; the four modules
  currently behind `[[tool.mypy.overrides]] ignore_errors = true`
  (`core`, `menubar_app`, `agent_log_parser`, `fs_watcher`) start
  losing their override, one per sub-PR.
- **Track 2c — mypy `disallow_untyped_defs = true`**: final
  ratchet. By this point the overrides should be empty.

**LOC**: ~600 across the three sub-PRs.
**Risk**: medium — type errors surface real bugs; budget for 1–2
follow-up patches.

### Track 3 — Bundled small UX wins (Tier 2)

Ship as a single PR — each piece is too small for its own PR.

| Piece | Source | LOC |
|---|---|---|
| `notifications.context_level: minimal/standard/full` opt-in for git.remote in alerts | ADR 0007 §D7 | ~50 |
| `sentinel doctor --cleanup-backups --keep N` | ADR 0006 §D5 | ~80 |
| mtime-based ProjectContext invalidation on `.git/HEAD` change | ADR 0007 §D4 | ~30 |

| `install.sh` deprecation policy decision (Q3 follow-up) — depending on user signal: doc-only nudge, or deprecation warning in the script itself | Q3 | ~doc |

**LOC**: ~210 + tests + docs.
**Risk**: low — additive, no schema breaks.

## ADRs to write at v0.9 freeze

Two new ADRs anticipated; one mini-amendment.

- **ADR 0008 — Notification Context Level** (Track 3, the
  `notifications.context_level` toggle). Documents the three
  values, default (= `standard`), the per-channel rendering rules,
  and the revision of ADR 0007 §D7's "audit-log-only forever"
  default for `git.remote`.
- **ADR 0009 — Backup Retention Policy** (Track 3, the
  `--cleanup-backups --keep N` knob). Documents the cleanup
  contract, default `--keep` value, idempotency, dry-run flag.
- **Mini-amendment to ADR 0001 (HostContext)**: D2 reload-table
  row for the `mtime` invalidation behavior of ProjectContext.
  Not a supersede — additive.

Track 1 (perf) and Track 2 (mypy ratchet) need no new ADRs — they
are pure code/tooling work against existing contracts.

## Decisions deferred to the v0.9 kickoff conversation

Pick exactly **one** from §B before starting Track 3.

- **B1**: Should Track 3 also include **Cursor / VS Code Continue
  session metadata** (Tier 3 §I)? Adds ~200 LOC and a parser
  module; depends on whether a Cursor user has filed a request
  by then. Default: skip in v0.9 unless someone asked.
- **B2**: Should Track 3 also include **net_tracker session
  attribution via cross-collector correlation** (ADR 0007 §D5
  follow-up)? Adds ~250 LOC + a small ADR. Net forensics win, but
  only meaningful once Tier 4 dashboard work starts consuming the
  audit log analytically. Default: skip in v0.9, revisit when
  Tier 4 starts.
- **B3**: Should v0.9 add a **`Cross-event correlation`** primitive
  (Tier 3 §H)? "this typosquat command followed an unknown-host
  network connection 30s earlier" — high security value, ~500 LOC,
  needs a fresh ADR for the in-memory window vs. SQLite question.
  Default: skip in v0.9, revisit alongside Tier 4 dashboard
  decision.

The defaults above leave v0.9 lean: Tracks 1 + 2 + 3 only. Which
gives 3 weeks of focused work and a clean handoff to whatever Tier 4
direction the user picks for v0.10.

## Backlog (Tier 4 — not for v0.9, captured for the next big
   decision conversation)

- **K. Dashboard MVP** (web UI). README "Phase 2" promise.
  SQLite + FastAPI/HTMX, ship as a separate package
  (`sentinel-dashboard`). 4–6 weeks. Best aligned with the
  "audit log is an asset" thesis from the original review.
- **L. Cross-platform Linux**. Q4 = "추후 고려". FSEvents →
  inotify, lsof → ss, launchd → systemd. ~4–6 weeks of
  refactoring plus a new test matrix. Doubles potential user
  base.
- **M. Pro 분기 첫 기능**. Q5 = "고려". ADR 0004 (Pro Optionality)
  already preserves the design seams — what's missing is a
  concrete first Pro feature. Candidates: cloud audit forwarding,
  team-managed blocklist, paid notification channels (PagerDuty,
  Opsgenie), longer-than-90-day retention. Decision blocks: no
  business model, no licensing model, no separate-package vs.
  same-package call. Cannot start until at least the first feature
  is named.

## Resume instructions

When picking this up:
1. Read this proposal in full.
2. Check that nothing in §A inventory is stale (run `gh pr list`
   and look for v0.9-tagged work that may have been started ad-hoc).
3. Confirm scope with the user — especially the §B decisions
   (B1/B2/B3 default-to-skip) and the §K/L/M Tier 4 timing.
4. Open the freeze PR (Tracks order: ADRs first, then implementation
   PRs in the §"Suggested v0.9 sequencing" order).
5. Mirror the v0.8 PR pattern: agent → verify → review → fix →
   commit → push → PR → user merges → next track.

## Out-of-scope clarifications

- **No release rush**: v0.9 is a polish release. Ship when the work
  is clean, not by a calendar date. Three weeks is an estimate, not
  a deadline.
- **No SQLite dependency** in Track 1. If `update_event_by_id` O(N)
  is in the profile top-3, the answer is an in-memory index — not a
  database engine. SQLite is reserved for Tier 4 dashboard work
  per the v0.7 Track A SQLite-rejection precedent.
- **No new collector** in v0.9. Cursor parser is §B1 (decision
  deferred).
